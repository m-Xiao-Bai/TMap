# TMap — 多城市轨道交通智能出行平台

> 全栈独立开发 | Spring Boot 3 + Vue 3 + 微信小程序 | 2024-2025

---

## 一、项目概述

TMap 是一个覆盖多城市、多线路的轨道交通出行平台，提供地铁线路查询、智能路径规划、在线购票（含 QR 码验票）、以及基于 LLM 的对话式路线规划 Agent。项目采用前后端分离架构，包含三个前端应用和一个 Java 后端服务。

**核心数据规模：** 多国家 / 多城市 / 数百条地铁线路 / 数千个站点的路网图数据

## 四、核心亮点与技术难点（后端重点）

### 亮点一：基于 JSON 邻接表的地铁图数据建模

**难点：** 地铁网络是一个典型的图结构——站点是节点，站点间的连接是边。一个站点可能属于多条线路（换乘站），在不同线路上有不同的前后邻居。传统方案需要站点表、线路表、站点-线路关联表、边表四张表，查询时需要多次 JOIN。

**方案：** 将图拓扑直接存储在站点实体的 JSON 数组字段中，而非独立的边表：

```java
// MetroStation 实体核心字段
Long id;                         // 雪花 ID
String stationName;              // 站名
String stationNameEn;            // 英文名
String stationAlias;             // 别名（支持模糊匹配）
BigDecimal longitude, latitude;  // 经纬度
Integer isTransfer;              // 是否换乘站 (0/1)
String lineIds;                  // JSON 数组: ["101","102"] -- 所属线路 ID
String lineNames;                // JSON 数组: ["1号线","2号线"] -- 所属线路名
String prevStationIds;           // JSON 数组: ["501","503"] -- 前驱站点 ID（按线路索引）
String nextStationIds;           // JSON 数组: ["502","504"] -- 后继站点 ID（按线路索引）
String prevStationDistances;     // JSON 数组: ["1.2","0.8"] -- 前驱距离 (km)
String nextStationDistances;     // JSON 数组: ["1.5","1.1"] -- 后继距离 (km)
```

**优势：**
- 用单表即可表达多线路换乘关系，无需中间关联表，查询时一次加载全部所需信息
- 支持按线路索引定位邻居，天然区分不同线路上的邻居关系
- 数据导入时可一次性写入站点及其邻居关系

**图构建代码（PathPlanningService）：**
```java
Map<Long, List<Long>> graph = new HashMap<>();
for (MetroStation s : stations) {
    graph.put(s.getId(), new ArrayList<>());
    graph.get(s.getId()).addAll(parseIdList(s.getPrevStationIds()));
    graph.get(s.getId()).addAll(parseIdList(s.getNextStationIds()));
}
```

**线路排序算法（MetroStationServiceImpl.getOrderedStationsByLineId）：**
1. 用 MySQL `JSON_CONTAINS` 查询属于目标线路的所有运营中站点
2. 通过 `getSameLineNeighbors()` 构建仅限同线路的邻接子图
3. 找度为 1 的端点站，从端点 BFS 遍历得到有序站列表
4. 处理分支线（找最长路径）和环形线（无端点时按 ID 排序）
5. 无连接数据时回退到开通日期排序

### 亮点二：BFS 最短路径 + 双层内存图缓存

**难点：** 每次路径规划都需要加载全量站点数据构建图，数据库查询开销大。且存在两处独立的 BFS 实现（PathPlanningService 和 TicketOrderServiceImpl），需要各自缓存。

**方案：**

**1. 本地内存图缓存（TicketOrderServiceImpl，5 分钟 TTL + 双重检查锁）：**
```java
private volatile List<MetroStation> cachedStations;
private volatile Map<Long, MetroStation> cachedStationMap;
private volatile Map<Long, List<Long>> cachedGraph;
private volatile long cacheTimestamp;
private static final long STATION_CACHE_TTL_MS = 5 * 60 * 1000;

// 双重检查锁保证线程安全
List<MetroStation> getCachedStations() {
    if (cachedStations == null || System.currentTimeMillis() - cacheTimestamp > STATION_CACHE_TTL_MS) {
        synchronized (this) {
            if (cachedStations == null || System.currentTimeMillis() - cacheTimestamp > STATION_CACHE_TTL_MS) {
                refreshStationCache();
            }
        }
    }
    return cachedStations;
}
```

**2. Redis 缓存（MetroStationServiceImpl，按城市缓存，24 小时 TTL）：**
- 读操作先查 Redis，miss 则查库并回填
- 写操作后主动调用 `deleteMetroStationListCache(cityId)` 失效
- 处理 Redis 反序列化问题：缓存对象可能是 `LinkedHashMap` 而非 `MetroStation`，使用 Jackson `convertValue()` 重新转换

**3. BFS 最短路径算法（两处相同实现）：**
```java
private List<Long> bfsShortestPath(Long startId, Long endId, Map<Long, List<Long>> graph) {
    Queue<Long> queue = new LinkedList<>();
    Map<Long, Long> parent = new HashMap<>();
    Set<Long> visited = new HashSet<>();
    queue.offer(startId);
    visited.add(startId);
    parent.put(startId, null);
    while (!queue.isEmpty()) {
        Long current = queue.poll();
        if (current.equals(endId)) return reconstructPath(parent, endId);
        for (Long neighbor : graph.getOrDefault(current, Collections.emptyList())) {
            if (!visited.contains(neighbor)) {
                visited.add(neighbor);
                parent.put(neighbor, current);
                queue.offer(neighbor);
            }
        }
    }
    return Collections.emptyList(); // 无路径
}
```

**4. 换乘检测与路线详情构建（PathPlanningService.buildRoutePlan）：**
- BFS 得到站点序列后，遍历路径调用 `determineLine()` 判断当前所在线路
- `determineLine()` 逻辑：当前线路仍在站的 lineIds 中 -> 继续；否则找当前站和下一站的公共线路；回退到站的第一条线路
- 线路切换时记录 `TransferInfo`（换乘站、从哪线到哪线）
- 距离计算：优先使用站点存储的真实 `nextStationDistances`，回退到 `kmPerStop`（默认 1.8km）
- 时间计算：`max(minMinutes, stationCount * minutesPerStop)`，参数可配置
- 票价计算：`ticket.price_tiers` 配置的阶梯票价（默认：3站2元、6站3元、9站4元、12站5元、18站6元、999站7元）

### 亮点三：5 节点 AI Agent 流水线（责任链架构）

**难点：** 用户的自然语言查询（如"从小蛮腰怎么坐地铁去白云机场"）需要经过意图识别、地名消歧、城市匹配、路径规划、回复生成等多个步骤，每一步都可能失败或需要降级。且需要支持流式输出、会话管理、取消操作。

**整体架构：**
```
Client (WebSocket) -> AgentWebSocketHandler -> RouteAgentEngine -> [5 AgentNodes] -> LLM API
```

**AgentNode 接口（最小契约）：**
```java
public interface AgentNode {
    void execute(AgentContext ctx, Consumer<Object> push) throws Exception;
    default String name() { return getClass().getSimpleName(); }
}
```

**AgentContext（共享可变状态袋，`@Data @Builder`）：** 在 5 个节点间传递的上下文对象，字段分组：
- WebSocket 信息：`wsSessionId`, `userId`, `anonToken`, `chatSessionId`, `clientIp`
- 用户输入：`userMessage`, `lat`, `lng`, `inputMethod`
- Node 1 输出：`intent`, `slotFrom`, `slotTo`, `llmInferredCity`
- Node 2 输出：`fromGeo` (LocationVO), `toGeo` (LocationVO)
- Node 3 输出：`fromCityId`, `fromCityName`, `toCityId`, `toCityName`, `scenario`
- Node 4 输出：`routePlan` (RoutePlanVO), `fromStationDistKm`, `toStationDistKm`
- Node 5 输出：`assistantReply`, `chips`
- 流控：`shortCircuit` (默认 false), `runId` (默认 UUID)
- Token 统计：`tokensIn`, `tokensOut`

**RouteAgentEngine（编排器，`@Service`）：**

线程模型：
- `Executors.newCachedThreadPool` + 自定义线程工厂（daemon 线程，名 `"agent-worker"`）
- `ConcurrentMap<String, Future<?>> running` 跟踪在途任务，支持取消
- `runAsync()` 立即返回 `runId`，流水线在后台线程异步执行

流水线执行（`runPipeline`）：
```java
AgentNode[] pipeline = {intentExtractNode, resolveLocationNode, matchCityNode, pathPlanNode, replyGenerateNode};
for (AgentNode node : pipeline) {
    long t0 = System.currentTimeMillis();
    node.execute(ctx, push);
    logNode(ctx, node, t0, true, null);
    if (ctx.shouldShortCircuit()) {
        // 短路处理：如果还没生成回复，强制执行 ReplyGenerateNode
        if (isBlank(ctx.getAssistantReply()) && !(node instanceof ReplyGenerateNode)) {
            replyGenerateNode.execute(ctx, push); // 生成确定性模板回复
        }
        break;
    }
}
```

持久化与统计：
- `saveUserMessage()` -- 保存用户消息到 `chat_message` 表，更新会话消息计数和自动标题
- `saveAssistantMessage()` -- 保存助手回复（含 intent、extras JSON 中的路线卡片/chips/scenario、LLM 模型、延迟）
- `upsertTokenUsage()` -- 按 (日期, 模型, 用户) 聚合 token 用量和费用（分），插入或累加 `agent_token_usage` 表
- `logNode()` -- 每个节点执行后记录 `chat_intent_log`（节点名、成功/失败、延迟、traceId）

错误处理：所有 DB 操作包裹 try-catch + `log.warn`（fail-soft），节点执行异常向上传播到 `runAsync` 的 catch 块，推送 error 事件给客户端。

**5 个节点详解：**

#### Node 1: IntentExtractNode（意图提取）

- 构建 system prompt 指示 LLM 提取意图、出发地、目的地、推断城市
- **Prompt 注入防御：** 用户输入用 `[USER_INPUT]...[/USER_INPUT]` 标记包裹，标签本身在用户输入中出现时转义
- LLM 调用：同步模式，temperature=0.1，maxTokens=200（硬编码，省 token）
- 解析 JSON 响应：提取 `intent`、`slots.from`、`slots.to`、`slots.city`
- **降级策略：** 异常时将整个用户消息当作 `slotTo`（乐观回退——假设用户描述了目的地）
- 短路：from 和 to 都为 null 时短路

#### Node 2: ResolveLocationNode（地名解析）

- 城市提示推断优先链：`llmInferredCity` > 浏览器定位逆地理编码 > session `lastCity` > IP 定位
- `safeGeocode()` 两阶段：先带城市约束编码，失败则不带城市约束
- 出发地解析失败时的降级：浏览器经纬度逆地理编码 -> IP 定位
- 目的地编码时，使用已解析出发地的城市作为新提示（升级提示精度）
- **session 记忆：** 将解析出的城市写回 `chat_session.extras.lastCity`（手工 JSON 拼接，避免 Jackson 依赖）
- 短路：toGeo 为空且 slotTo 为空时，scenario=`MISSING_DEST`

#### Node 3: MatchCityNode（城市匹配）

- 最高优先：LLM 推断城市 -> 直接查 DB（LIKE 匹配 cityName/cityNameEn/cityAlias，statusCode=3）
- 高德地理编码城市匹配
- 站名反查城市：剥离 "地铁站/火车站/站" 后缀 -> 精确匹配 -> LIKE 匹配 -> **兜底：遍历所有已开通城市检查输入是否包含城市名**（处理"南昌机场"这种嵌入式城市名）
- 场景判定：两个都空 -> `NO_METRO`；两个都有但不同 -> `CROSS_CITY`；否则 -> `SAME_CITY`
- 短路：`NO_METRO` 和 `CROSS_CITY` 场景短路

#### Node 4: PathPlanNode（路径规划，最复杂节点）

**5 级候选收集（collectCandidates）：**

1. **LLM-RAG（最高优先级）：** `collectFromLlmDirect()` -- 加载城市全量站名列表 -> 调用 `llmLocationResolver.pickNearestFromStationList(slotName, cityName, availableStations)` -> LLM 从白名单中选 1-3 个最近站（约束在真实站名列表中，防幻觉）-> 防御性精确/模糊匹配验证 -> Haversine 距离计算
2. **精确名称匹配：** `exactMatch()` -- 剥离 "地铁站/火车站/站" 后缀 -> 查 stationName（原始+清理后）、stationNameEn、stationAlias
3. **地理 TopN：** `nearbyStationService.findLocalCandidates(lat, lng, cityId)` -- Haversine 距离排序取前 N
4. **模糊名称匹配：** `likeMatch()` -- SQL LIKE 查询，返回最短名称匹配（最精确），距离 > 10km 的丢弃
5. **LLM 标准名翻译：** `llmLocationResolver.suggestStandardNames()` -- 如"小蛮腰" -> "广州塔"，对每个翻译结果重新地理编码+基本匹配

**N x N 组合 BFS 搜索（pickBestCombination）：**
```java
for (NearbyStation start : starts) {
    for (NearbyStation end : ends) {
        RoutePlanVO plan = pathPlanningService.planRoute(start.id, end.id, cityId);
        double cost = legMinutes(start.distKm) + plan.stationCount * 2.5 + legMinutes(end.distKm);
        // 选 cost 最小的组合
    }
}
```

**末段距离分级速度模型（legMinutes）：**
| 距离 | 模式 | 速度 |
|------|------|------|
| < 0.5km | 忽略（在站口） | 0 |
| 0.5-1.5km | 步行 | 5 km/h |
| 1.5-3.0km | 骑行 | 15 km/h |
| > 3.0km | 打车/公交 | 30 km/h |

**OSM 降级（tryOsmHint）：** 本地无候选时查询 OpenStreetMap Overpass API（地铁站/轻轨站/地铁口），结果仅记录日志（不参与 BFS，因为无本地站 ID）。

#### Node 5: ReplyGenerateNode（回复生成）

**两种输出模式：**

1. **确定性模板（7 种场景）：** `CROSS_CITY` / `NO_METRO` / `SAME_STATION` / `NO_ROUTE` / `START_NOT_FOUND` / `END_NOT_FOUND` / `NO_STATIONS_FOUND` -- 不调用 LLM，直接推送模板文本 + 场景化 chips
2. **LLM 流式生成：** temperature=0.3，SSE 流式推送每个 delta 到前端

**System Prompt 构建：** 从 `agent.prompt.system` 配置读取模板，替换 `{{userLogged}}`、`{{selectedCity}}`、`{{routePlanSummary}}` 占位符。

**RAG 知识注入：** 调用 `ragService.retrieveForContext(userMessage, cityId, stationIds)`，将检索到的知识作为 system message 插入消息列表。

**末段交通建议（buildLegHint）：**
| 距离 | 建议 |
|------|------|
| < 0.5km | 无（在站口） |
| 0.5-1.5km | 步行约 N 分钟 |
| 1.5-3.0km | 骑共享单车约 N 分钟 |
| 3.0-5.0km | 打车/公交约 N 分钟 |
| > 5km | 警告位置不便，建议直接打车 |

**流式失败降级：** LLM 流式调用异常时，回退到 `buildFallbackReply()` 推送确定性文本。

**上下文敏感 Chips（buildChips）：** 根据场景返回不同的快捷回复按钮标签，如成功路线返回 "是的，帮我下单" / "换个方案" / "为什么这样换"。

### 亮点四：多候选 N x N 全局最优路径规划

（详见亮点三 Node 4 部分，此处补充 LLM 防幻觉机制）

**LLM 防幻觉三重机制：**
1. **RAG 约束（pickNearestFromStationList）：** 将城市真实站名列表（最多 300 个，约 1200 token）喂给 LLM，严格要求回答必须从列表中选择
2. **后校验：** LLM 返回的站名必须在数据库中精确/模糊匹配，不匹配则丢弃
3. **降级：** LLM 路径不可用时回退到精确匹配 + 地理 TopN + 模糊匹配

### 亮点五：三层缓存架构（ConcurrentHashMap + Redis + 本地内存）

| 缓存层 | 内容 | TTL | 失效策略 |
|--------|------|-----|---------|
| **ConcurrentHashMap** | system_config 全量配置 | 永久（启动加载，更新时 refreshCache） | 写操作后原子替换 |
| **Redis** | 国家/城市/线路/站点列表 | 可配置（默认 24h） | 写操作主动删除 + 管理端手动清除 |
| **Redis** | JWT Token | 24h | 登录时覆盖 |
| **Redis** | 验证码（图片/邮件） | 5min | 使用后删除 |
| **Redis** | 登录失败计数 | 30min | 自动过期 |
| **Redis** | IP 限流计数 | 60s | 自动过期 |
| **本地内存 (volatile)** | 站点图邻接表 + 站点 Map | 5min | 双重检查锁 + 写操作调用 invalidateStationCache() |

**SystemConfigServiceImpl 配置缓存：** `ConcurrentHashMap<String, String>` 全量缓存，`@PostConstruct` 启动加载，`updateConfigs()` 后原子替换。支持类型化获取：`getConfigInt`、`getConfigObject`（Jackson 反序列化）、`getRaw`（解密 + 清理不可见字符）。

**Redis 序列化：** `Jackson2JsonRedisSerializer` + `JavaTimeModule`，`RedisTemplateWrapper` 封装 String/Counter/Hash/Set 操作，`increment()` 方法首次增量时自动设置 TTL（限流基础设施核心）。

**缓存管理 API：** `GET /manage/cache/status`（各类缓存计数）+ `DELETE /manage/cache/clear/{category}`（按类清除），支持运行时运维。

### 亮点六：全链路安全体系

**HTTP 请求拦截链（按执行顺序）：**

```
请求 -> RateLimitInterceptor -> TokenInterceptor -> Controller -> @RequireRole AOP -> ExceptionLogAspect -> GlobalExceptionHandler
```

**1. RateLimitInterceptor（IP 滑动窗口限流）：**
- Redis key: `rate_limit:<ip>`，TTL=windowSeconds（默认 60s），最大请求数=maxRequests（默认 60）
- IP 解析优先链：`X-Forwarded-For`（第一个） > `X-Real-IP` > `request.getRemoteAddr()`
- 超限返回 HTTP 429 + JSON 错误
- OPTIONS 和 /error 路径始终放行

**2. TokenInterceptor（JWT + Redis 双校验）：**
- 解析 `Authorization: Bearer <token>` 头
- `jwtUtil.validateToken(token)` 校验签名和过期
- `redisUtils.getToken(userId)` 二次校验——实现 token 吊销：用户重新登录时旧 token 在 Redis 中被覆盖
- 设置 `userId` 和 `roleCode` 请求属性

**3. RBAC（@RequireRole + RoleCheckAspect AOP）：**
```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireRole {
    int[] value() default {}; // 允许的角色码
}
```
AOP `@Around` 切面从 request attribute 读取 roleCode，与注解的 `int[] value()` 比较，不匹配抛 `BusinessException(ErrorCode.FORBIDDEN)`。

**4. ExceptionLogAspect（全局异常日志）：**
- `@AfterThrowing(pointcut = "execution(* com.mu..*(..))")` -- 拦截 com.mu 包下所有方法
- 记录类名、方法名、异常类型、异常消息、完整堆栈

**WebSocket 安全链：**

```
HTTP Upgrade -> AgentWebSocketInterceptor (握手鉴权) -> AgentWebSocketHandler (运行时限流)
```

**AgentWebSocketInterceptor：**
- 从 query param 提取 `token`、`anon`、`sessionId`
- 有 token：校验 JWT + Redis，失败则降级到匿名
- 无 token 且无 anon：拒绝握手
- 设置 `userId`、`anonToken`、`sessionId`、`clientIp` 属性

**AgentWebSocketHandler 四重限流：**
1. **匿名每日限额：** Redis key `agent:anon:daily:<anonToken>:<date>`，TTL 25h，默认 10 条/天
2. **登录用户每分钟限流：** Redis key `agent:rl:user:<userId>:<minute>`，TTL 65s，默认 30 次/分钟
3. **IP 每分钟限流：** Redis key `agent:rl:ip:<ip>:<minute>`，TTL 65s，默认 60 次/分钟
4. **敏感词过滤：** 配置 `agent.security.forbidden_words`，大小写不敏感子串匹配

**并发连接限制：** `AgentSessionRegistry` 使用 4 个 ConcurrentHashMap 跟踪 session/ws/owner 映射，`countByOwner()` 检查每用户连接数（默认 3），超限关闭自定义状态码 4029。

**其他安全措施：**
- BCrypt 密码加密
- 登录失败锁定：连续 10 次失败锁定 30 分钟（Redis 计数器）
- 图片/邮箱验证码，Redis 存储 + 5 分钟 TTL
- LLM API Key / 高德 API Key 支持 `enc:` 前缀 AES 加密存储，`getRaw()` 解密 + 清理零宽字符
- Prompt 注入防御：`safeUserInput()` 转义 `[USER_INPUT]` 标记

### 亮点七：订单生命周期与原子操作

**6 状态有限状态机：**
```
0(待支付) --> 1(已支付) --> 2(已使用) --> [终态]
    |             |
    v             v
  3(已过期)     5(审核中) --> 4(已退款)
                  |
                  v
                1(驳回)
```

**合法状态转换（isValidStatusTransition）：**
- 0 -> 1 (支付), 0 -> 3 (过期)
- 1 -> 2 (使用), 1 -> 3 (过期), 1 -> 5 (申请退款)
- 5 -> 1 (驳回), 5 -> 4 (批准退款)
- 2, 3, 4 为终态

**原子支付（payOrder）：**
```java
LambdaUpdateWrapper<TicketOrder> wrapper = new LambdaUpdateWrapper<>();
wrapper.eq(TicketOrder::getId, orderId)
       .eq(TicketOrder::getStatus, 0)  // CAS 条件
       .set(TicketOrder::getStatus, 1)
       .set(TicketOrder::getPayTime, LocalDateTime.now());
boolean success = update(wrapper); // affected rows = 0 表示状态已变更
```

**原子验票（verifyQrCode）：** 同理，`status=1 -> status=2` 的 CAS 更新，防止并发双扫。

**定时过期（TicketExpireSchedule）：** `@Scheduled(fixedRate = 5 * 60 * 1000)` 每 5 分钟扫描，过期未支付订单和 QR 码过期的已支付订单，批量更新为 status=3，逐个发送系统消息。

**批量购票：** 单次最多 10 张，共享 BFS 路径计算结果，`saveBatch()` 批量插入，生成唯一订单号（`TK` + 时间戳 + 6 位随机数）和 UUID QR 码。

## 八、面试高频问题准备

### Q: 为什么用 BFS 而不是 Dijkstra？
A: 地铁出行场景中，用户最关心的是换乘次数最少、站点数最少，而非物理距离最短。BFS 天然求最少站点数路径（无权图最短路），时间复杂度 O(V+E)。虽然边权（距离）存在于数据中，但仅用于事后计算距离和时间，不参与路径搜索。如果未来需要加权最短路（如考虑换乘惩罚），可以升级为 Dijkstra 或 A*。

### Q: JSON 存储图拓扑 vs 独立边表，怎么选？
A: JSON 方案的优势：（1）单表查询即可获得站点的全部拓扑信息，无需 JOIN；（2）天然支持多线路索引——同一站点在线路 A 和线路 B 上有不同的邻居，通过 JSON 数组索引区分；（3）数据导入时可一次性写入站点及其邻居关系。劣势是不利于复杂图查询（如全图连通分量、最小生成树），但对本项目"给定起终点求最短路径"的场景完全适用。如果数据规模增长到百万站点，可以考虑 Neo4j 等图数据库。

### Q: Agent 流水线如何处理失败？
A: 五重保障：（1）每个节点内部 try-catch，异常时设置降级策略（如 IntentExtractNode 异常时将整个用户消息当作目的地）；（2）节点间短路机制，任意节点可设置 `shortCircuit=true` 跳过后续节点；（3）引擎层兜底：短路时如果还没生成回复，强制执行 ReplyGenerateNode 生成确定性模板；（4）ReplyGenerateNode 自身的 LLM 流式失败降级到模板回复；（5）所有 DB 操作 fail-soft（try-catch + warn 日志），不阻塞主流程。

### Q: 如何保证订单不被重复支付？
A: 使用带状态条件的 CAS 更新：`UPDATE ticket_order SET status=1, pay_time=now WHERE id=? AND status=0`。如果 affected rows = 0，说明状态已变更（已支付/已过期），抛出 `TICKET_ORDER_STATUS_ERROR` 异常。验票同理：`status=1 -> status=2` 的原子转换。这比先查后改的方案更安全，因为查和改之间没有时间窗口。

### Q: 缓存一致性如何保证？
A: 三层缓存各有策略：（1）ConcurrentHashMap 配置缓存——写操作后原子替换整个 Map（`clear()` + `putAll()`）；（2）Redis 业务缓存——Cache-Aside 模式，写操作先更新 DB 再主动删除缓存 key；（3）本地 volatile 图缓存——写操作调用 `invalidateStationCache()` 置空所有字段 + 重置时间戳。管理端提供手动清除 API 应对紧急场景。

### Q: LLM 如何防止幻觉（hallucination）？
A: 三重机制：（1）RAG 约束——`pickNearestFromStationList()` 将城市真实站名列表（最多 300 个）作为上下文喂给 LLM，严格要求回答必须从中选择，不允许编造；（2）后校验——LLM 返回的站名必须在数据库中精确/模糊匹配，不匹配则丢弃；（3）降级——LLM 路径不可用时回退到精确匹配 + 地理 TopN + 模糊匹配，保证总有候选结果。

### Q: WebSocket 如何保证安全性？
A: 四层防护：（1）握手鉴权——`AgentWebSocketInterceptor` 校验 JWT + Redis Token，无认证拒绝握手；（2）运行时限流——匿名每日限额（Redis 计数器）+ 登录用户每分钟限流 + IP 每分钟限流；（3）并发连接限制——`AgentSessionRegistry` 跟踪每用户连接数，超限关闭 4029；（4）内容过滤——敏感词大小写不敏感子串匹配。限流采用 fail-open 策略（Redis 异常时放行），避免 Redis 故障导致全部用户被锁。

### Q: 如何设计 LLM 调用的可观测性？
A: 三个维度：（1）`ChatIntentLog` -- 每个节点执行后记录节点名、成功/失败、错误消息、延迟、traceId；（2）`AgentTokenUsage` -- 按 (日期, 模型, 用户) 聚合 input/output token 数和费用（分），支持成本监控；（3）`AgentContext.latencyMs` -- 端到端延迟追踪。MDC 设置 `traceId` 和 `sessionId` 支持日志链路追踪。

### Q: 前端三种地图模式的设计考量？
A: 各有适用场景：（1）地理视图（Leaflet + 高德瓦片）-- 真实地理坐标，适合了解站点实际位置；（2）示意图（Leaflet 无瓦片）-- 白色背景 + BFS 路径叠加 + 动画虚线，适合路线规划结果展示；（3）拓扑图（SVG + D3-zoom）-- 自动布局算法（保留地理角度 + 最小间距碰撞解决 + 8 方向标签放置 + 贝塞尔圆角），适合全局线路概览。用户端默认示意图，管理端三种都支持。
